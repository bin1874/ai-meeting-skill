This is a shortened example of the kind of final report `ai-meeting` writes to
`meetings/<id>/synthesis/final.md`.

```md
# AI Meeting 结论

## 最终建议
建议选择：修改后的方案

置信度：中

一句话结论：Proceed with the feature only as a small, permission-safe MVP; defer advanced administration until real team usage proves demand.

## 核心理由
1. The proposal has clear user value for teams that need shared work, but the full scope creates avoidable permission and migration risk.
2. A smaller MVP can validate demand without committing to a complex enterprise workspace model.
3. The largest unresolved risk is not UI implementation; it is access control correctness and migration safety.

## 改进后的方案
Ship a limited team workspace MVP:

- Create and invite team members.
- Support owner/admin/member roles.
- Keep existing personal workspace behavior unchanged.
- Add audit logging for membership and permission changes.
- Measure activation, invite acceptance, and shared project usage.

## 不建议采用的方案
Do not ship the full enterprise workspace system in the next release. It adds billing, nested roles, transfer flows, and migration logic before demand is validated.

## 反对意见
1. A limited MVP may not satisfy enterprise prospects that need SSO, SCIM, or detailed RBAC.
2. Delaying advanced controls could require rework if the MVP model is too simplistic.

## 最大风险
| 风险 | 严重度 | 可能性 | 缓解方式 |
|---|---:|---:|---|
| Permission bug exposes private projects | 高 | 中 | Keep the role model small, add permission matrix tests, and require audit logs |
| Migration confuses existing users | 中 | 中 | Make personal workspace unchanged and opt-in only |

## 各 Agent 立场
| Agent | Provider | 立场 | 核心观点 | 置信度 |
|---|---|---|---|---:|
| Builder | Codex | 修改 | MVP is achievable in the release window | 0.78 |
| Critic | Claude | 修改 | Full scope is too risky without permission proof | 0.84 |
| Architect | Codex | 修改 | Keep workspace boundaries explicit and testable | 0.80 |

## 主要争议
1. Whether enterprise value requires advanced controls immediately.
2. Whether permission complexity should block the entire feature.

## 已达成共识
1. The full proposal is too large for the next release.
2. A smaller MVP is useful if permission boundaries are explicit.
3. The final decision needs evidence from real team adoption.

## 证据缺口
1. No production telemetry proves how many users need team workspaces.
2. No permission matrix test plan has been reviewed.
3. Large design files, if truncated in prompts, must be treated as partial evidence.

## 待验证问题
1. What percentage of active users invite at least one collaborator?
2. Which roles are actually needed in the first 30 days?
3. Can support handle workspace migration questions?

### 1 周内
```
